# PDF Reporting Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Stand up server-side PDF generation for the legally-mandated reports (jegyzőkönyv, éves elszámolás, rezsi statement) and the marketing-promised quick exports (finance summary, voting results, audit log slice). Locale-aware (HU number/date formatting), retained per the same rules as the source data, hashable for legal use.

**Architecture:** `@react-pdf/renderer` for templated reports (JSX templates, no Chromium runtime, fast cold starts). PDFs generated by BullMQ worker jobs and stored in R2 keyed by `buildingId/kind/period/contentHash`. Inline rendering allowed for small reports (< 2 pages, < 50 rows) with a 10s timeout. Five phases.

**Tech Stack:** Next.js 15, Prisma, BullMQ on Redis, `@react-pdf/renderer`, `sharp` for embedded images. Cross-references the file-storage plan (R2) and audit-ui plan (audit slice export).

**Spec source:** Audit findings dated 2026-04-28. Cross-references: `docs/plans/2026-04-28-file-storage-strategy.md` (storage), `docs/plans/2026-04-27-roles-legal-alignment.md` (signature blocks).

**Non-goals:**
- Real PKI / qualified e-signature (NetLock or similar HU CA) — separate plan.
- Editable PDF forms.
- Excel exports — `xlsx` already a dep, used separately.
- DOCX exports.
- White-label / custom branding per building (Kezelő iroda tier feature, separate plan).
- AI-summarised reports.

## Legal Anchors

| Report | Source | Deadline | Retention |
|---|---|---|---|
| **Közgyűlés jegyzőkönyv** | Tht. § 39 | Within 8 days of meeting (delivery) | Life of condominium (Tht. § 44) |
| **Éves költségvetés és elszámolás** | Tht. § 47–48 | Approved by 31 May | 8 years (Számviteli tv. § 169) |
| **Annual rezsi statement** | Tht. § 43/A | To each owner by 15 February | 8 years |
| **Monthly utility (rezsicsökkentési kimutatás)** | Tht. § 43/A | Posted by 15th, displayed 45 days | 8 years |
| **Audit-log slice export** | GDPR Art. 15 (when user-requested) | On request | Per audit retention plan |

## Phase 1: Library + smallest report (vote result)

**Goal:** Smallest end-to-end. Install `@react-pdf/renderer`, set up the lib directory, render a vote-result PDF inline (no worker yet) as the canary.

- [ ] **Step 1: Add deps**
  ```bash
  npm install @react-pdf/renderer
  ```

- [ ] **Step 2: Embed fonts**. Add `Manrope`, `Space Grotesk`, `IBM Plex Mono` font files under `src/reports/lib/fonts/`. Register via `Font.register({ family: "Manrope", src: "..." })`. Use the same font stack as the design system so reports match the on-screen look.

- [ ] **Step 3: HU formatters** in `src/reports/lib/format.ts`:
  ```ts
  export const formatHUF = (v: number) => new Intl.NumberFormat("hu-HU", { style: "currency", currency: "HUF", maximumFractionDigits: 0 }).format(v);
  export const formatDate = (d: Date) => new Intl.DateTimeFormat("hu-HU", { dateStyle: "long" }).format(d);
  export const formatDateTime = (d: Date) => new Intl.DateTimeFormat("hu-HU", { dateStyle: "short", timeStyle: "short" }).format(d);
  export const formatShare = (s: Decimal) => `${(Number(s) * 100).toFixed(2)} %`;
  ```

- [ ] **Step 4: Template `vote-result.tsx`** — header (building name, vote title, period), three KPI cells (eligible shares / cast shares / quorum %), result table with options × votes × percentage, footer with hash + timestamp + page n/N.

- [ ] **Step 5: `src/reports/lib/generate.ts`**:
  ```ts
  export async function renderToBuffer<P>(Template: React.FC<P>, props: P): Promise<Buffer> {
    const stream = await pdf(<Template {...props} />).toBuffer();
    return stream;
  }
  ```

- [ ] **Step 6: Inline route** at `src/app/api/reports/vote/[voteId]/route.ts`:
  - Validate session + read access to the vote.
  - Load the vote + ballots + tally.
  - Render and stream the PDF.
  - 10 s timeout — if exceeded, push to worker (Phase 2).

- [ ] **Step 7: Commit**
  ```bash
  git commit -m "feat(reports): phase 1 — @react-pdf/renderer setup + vote-result canary"
  ```

## Phase 2: Worker-based generation + R2 storage + caching

**Goal:** Move heavy reports to BullMQ. Cache by content hash. Polling endpoint for the client.

- [ ] **Step 1: Schema add `GeneratedReport`**:
  ```prisma
  model GeneratedReport {
    id            String   @id @default(cuid())
    building      Building @relation(fields: [buildingId], references: [id])
    buildingId    String
    kind          String   // "minutes" | "budget" | "year-end-account" | "utility-statement" | "finance-summary" | "vote-result" | "audit-slice"
    period        String   // "2026" | "2026-04" | "vote-{voteId}" — opaque per kind
    contentHash   String   // SHA-256 of the PDF bytes
    storageKey    String   // R2 key
    generatedById String
    generatedBy   User     @relation(fields: [generatedById], references: [id])
    generatedAt   DateTime @default(now())
    isDraft       Boolean  @default(false)
    retainUntil   DateTime?
    @@unique([buildingId, kind, period, contentHash])
    @@index([buildingId, kind])
  }
  ```

- [ ] **Step 2: Generate route** at `src/app/api/reports/generate/route.ts`. Accepts `{ kind, period, params }`. Computes `contentHash = SHA-256(buildingId|kind|period|dataVersion)`. If a `GeneratedReport` row already exists with that hash, returns its `id`. Otherwise enqueues a BullMQ job and returns `{ jobId }`.

- [ ] **Step 3: Worker job** `worker/jobs/generate-report.ts`:
  - Loads the data for the report kind.
  - Renders via `renderToBuffer`.
  - Uploads to R2 under `reports/{buildingId}/{kind}/{period}/{contentHash}.pdf` (cross-ref file-storage plan).
  - Inserts `GeneratedReport` row with `retainUntil` computed by kind:
    - minutes → `null` (forever)
    - year-end-account, finance-summary, utility-statement → 8 years
    - vote-result → 6 years
    - audit-slice → 30 days

- [ ] **Step 4: Status route** at `src/app/api/reports/[id]/status/route.ts`. Returns `{ status: "queued" | "running" | "ready" | "error", progress?, downloadUrl? }`.

- [ ] **Step 5: Download route** at `src/app/api/reports/[id]/download/route.ts`. Validates RBAC by report kind (e.g. minutes are visible to all building members; finance-summary requires `view.building.finance` capability). Returns signed-GET URL from R2.

- [ ] **Step 6: Worker concurrency**. PDF generation can be CPU-spiky. Cap concurrency at 2 in the worker config; queue overflow is acceptable.

- [ ] **Step 7: Migration + commit**
  ```bash
  npx prisma migrate dev --name phase2-generated-report
  git commit -m "feat(reports): phase 2 — worker-based generation + R2 caching"
  ```

## Phase 4: Jegyzőkönyv (minutes) with signature block

**Goal:** Tht. § 39-compliant minutes PDF with signature block holding chair + two hitelesítő signatures. NOT real PKI — that's a separate plan.

- [ ] **Step 1: Schema add `MeetingMinutesSignature`**:
  ```prisma
  model MeetingMinutesSignature {
    id        String   @id @default(cuid())
    meeting   Meeting  @relation(fields: [meetingId], references: [id])
    meetingId String
    signer    User     @relation(fields: [signerId], references: [id])
    signerId  String
    role      String   // "CHAIR" | "AUTHENTICATOR_1" | "AUTHENTICATOR_2"
    signedAt  DateTime @default(now())
    ipAddress String?
    @@unique([meetingId, role])
  }
  ```

- [ ] **Step 2: Sign action**. After a meeting closes, the chair and two designated authenticators (`Meeting.authenticatorIds`) get a "Sign minutes" task. Action records the signature and timestamps. Once all three are present, the minutes are "executed".

- [ ] **Step 3: `minutes.tsx` template**. Sections:
  - Header: building name, meeting type, date, location
  - Attendance: list of present owners with ownership shares + total
  - Quorum confirmation
  - Each `Vote.határozat` with full text + tally + result + statutory majority verification
  - Resolutions adopted
  - Adjournment time
  - Signature block with three slots: Chair, Authenticator 1, Authenticator 2 — each filled with name + signedAt + IP if signed, else "(aláírásra vár)"
  - Page footer with SHA-256 hash + page n/N

- [ ] **Step 4: Compliance**. After all three signatures are present, regenerate the PDF (new `contentHash`); the prior unsigned version stays in storage with `isDraft = true`. Both rows persist forever (Tht. § 44).

- [ ] **Step 5: Commit**
  ```bash
  npx prisma migrate dev --name phase4-meeting-minutes-signature
  git commit -m "feat(reports): phase 4 — jegyzőkönyv with signature block (Tht. § 39)"
  ```

## Phase 5: Audit-log slice + DRAFT watermark + scheduled annual reports

**Goal:** Tie-in with audit-ui plan; DRAFT watermark for unapproved reports; scheduled cron for the annual rezsi statement.

- [ ] **Step 1: `audit-slice.tsx`**. Header: filter params, generated by, generated at, total row count. Table of audit rows (timestamp, actor, action, entity, short diff). Footer: HMAC-signed manifest hash (cross-ref audit-ui plan, Phase 4).

- [ ] **Step 2: DRAFT watermark**. CSS in templates: when `props.isDraft === true`, render a 45°-rotated "TERVEZET" watermark across each page in 20% opacity. Strip on approval (regenerates the PDF).

- [ ] **Step 3: Print fallback**. For each report kind, also expose `/[locale]/reports/[id]/print` as an HTML render. Same template props, but as React components in a print-optimized layout (with `@media print` CSS). Useful for users who want to print without the PDF round-trip.

- [ ] **Step 4: Scheduled annual rezsi worker** at `worker/jobs/scheduled-annual-rezsi.ts`. BullMQ cron: `0 8 1 2 *` (1 February at 08:00). For every active building:
  - Generate `kind = "utility-statement", period = "{year}"`.
  - Notify each owner via `Notification` (cross-ref real-time plan).
  - Tht. § 43/A deadline is 15 Feb — generating on 1 Feb gives 14 days for delivery.

- [ ] **Step 5: Scheduled annual budget reminder**. Similar cron `0 8 1 4 *` (1 April) — reminds the chair that the annual budget needs assembly approval by 31 May.

- [ ] **Step 6: Commit**
  ```bash
  git commit -m "feat(reports): phase 5 — audit-slice, DRAFT watermark, scheduled annual reports"
  ```

## Acceptance criteria

This plan is complete when:

1. A vote-result PDF for a vote with 50 ballots renders in **< 2 s**.
2. Re-requesting a report with the same `(buildingId, kind, period, dataVersion)` returns the cached row — no recomputation, no new R2 upload.
3. The jegyzőkönyv PDF embeds chair + two authenticator signatures with timestamps when all three have signed.
4. The annual rezsi statement is auto-generated by 1 Feb and a `Notification` is sent to each owner.
5. All HU number formatting uses `1 234 567 Ft` style (verified in template tests).
6. An unapproved budget report shows a `TERVEZET` watermark; the approved version does not.
7. Each PDF's footer contains a SHA-256 hash that matches the actual file's SHA-256.
8. A user lacking `view.building.finance` capability gets HTTP 403 from the finance-summary download route.
9. The print-fallback HTML route renders the same data as the PDF (visual-regression test).
10. Worker concurrency is capped at 2 for `generate-report` jobs (no PDF render starves the queue).

When all ten hold, PDF reporting is production-ready.
